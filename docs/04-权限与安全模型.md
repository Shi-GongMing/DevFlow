# 04 — 权限与安全模型

> **面试问题**："Agent 可以执行 bash 命令、修改文件，你怎么防止它干坏事？"
> **源文件**：[devflow/permissions/](../devflow/permissions/) (5个文件)
> **依赖关系**：agent.py 在执行每个工具前调用 `permission_checker.check()`

---

## 一、四层纵深防御模型

DevFlow 不靠单一道门，而是用四层叠加——每一层解决不同粒度的问题：

```
Layer 1: PermissionMode (全局模式)
  └── Layer 2: RuleEngine (持久化规则)
       └── Layer 3: DangerousCommandDetector (危险检测)
            └── Layer 4a: PathSandbox (路径白名单)
                 └── Layer 4b: _session_allowed (会话放行)
```

---

## 二、Layer 1：PermissionMode —— 全局行为矩阵

```python
class PermissionMode(str, Enum):
    DEFAULT = "default"               # 读取放行, 写/命令需确认
    ACCEPT_EDITS = "acceptEdits"      # 读/写放行, 命令需确认
    PLAN = "plan"                     # 只放行 plan-mode 白名单工具
    BYPASS = "bypassPermissions"      # 全部放行

# 决策矩阵
_MODE_MATRIX = {
    DEFAULT:      {"read": "allow", "write": "ask",   "command": "ask"},
    ACCEPT_EDITS: {"read": "allow", "write": "allow", "command": "ask"},
    PLAN:         {"read": "allow", "write": "ask",   "command": "ask"},
    BYPASS:       {"read": "allow", "write": "allow", "command": "allow"},
}

def mode_decide(mode, category) -> DecisionEffect:
    return _MODE_MATRIX[mode][category]
```

**设计 tradeoff：**
- **为什么 category 只需要 3 种？** read/write/command 是操作系统中经典的权限分类。更细的分类（如 network/process/fs）更精确，但工具开发者需要理解更多概念。3 种分类在"足够用"和"足够简单"之间取了平衡。
- **PLAN 模式的特殊白名单**：
  ```python
  _PLAN_MODE_ALLOWED_TOOLS = frozenset({
      "Agent", "ToolSearch", "AskUserQuestion", "ExitPlanMode"
  })
  ```
  PLAN 模式下只有这 4 个工具可用——Agent（创建子 Agent 做研究）、ToolSearch（搜索工具）、AskUserQuestion（向用户提问）、ExitPlanMode（退出计划模式）。其他工具（Bash、WriteFile 等）全部被拦截。

## 三、Layer 2：RuleEngine —— 持久化规则匹配

```python
class RuleEngine:
    def match(self, tool_name: str, content: str) -> Decision | None:
        # 遍历所有规则，返回第一个匹配的 allow/deny 决策
        # 本地规则（permissions.local.yaml）优先级高于项目规则（permissions.yaml）
```

**规则示例**：
```yaml
rules:
  - tool: Bash
    pattern: "npm test*"
    effect: allow
  - tool: WriteFile
    pattern: "*.md"
    effect: deny
  - tool: ReadFile
    pattern: ".env*"
    effect: deny
```

**为什么持久化规则有必要？** 用户可以在 yaml 中配置"永远允许 npm test"或"永远阻止读 .env 文件"。这些规则持久化后跨会话有效，不需要每次都手动确认。

**Layer 4b 的 add_session_allow 为什么比 Layer 2 优先级更高但又不持久化？**

```
用户点 "don't ask again" for "npm install express"
  → RuleEngine.append_local_rule(...)  # 写入 permissions.local.yaml
  → add_session_allow(...)            # 放入内存 set，本轮立即生效
```

两个操作同时发生：yaml 写入保证下次会话也有效，`_session_allowed` 保证本轮立即生效（不需要重新读文件）。

## 六、完整的权限检查流程

```python
# checker.py PermissionChecker.check()

def check(self, tool, arguments) -> Decision:
    # Step 1: 提取内容标识 (用于规则匹配)
    content = extract_content(tool.name, arguments)
    # 对于 Bash：提取命令文本
    # 对于 WriteFile：提取文件路径
    # 对于 ReadFile：提取文件路径

    # Step 2: 检查 Layer 4b 会话放行
    if f"{tool.name}:{content}" in self._session_allowed:
        return Decision(effect="allow", reason="session allow")

    # Step 3: 检查 Layer 2 规则引擎
    rule_decision = self.rule_engine.match(tool.name, content)
    if rule_decision:
        return rule_decision

    # Step 4: Plan 模式特殊处理
    if self.mode == PermissionMode.PLAN:
        if tool.name in _PLAN_MODE_ALLOWED_TOOLS:
            if tool.name == "ExitPlanMode":
                return Decision(effect="ask", reason="confirm plan exit")
            return Decision(effect="allow", reason="plan mode whitelist")
        if tool.category == "read":
            return Decision(effect="allow")
        return Decision(effect="deny", reason="plan mode: only read tools allowed")

    # Step 5: Layer 3 危险命令检测
    if tool.name == "Bash":
        if self.detector.is_dangerous(content):
            return Decision(effect="deny", reason="dangerous command detected")

    # Step 6: Layer 4a 路径沙箱
    if tool.category in ("write", "command"):
        if not self.sandbox.validate(arguments):
            return Decision(effect="deny", reason="path outside sandbox")

    # Step 7: Layer 1 模式决策
    effect = mode_decide(self.mode, tool.category)
    return Decision(effect=effect, reason=f"mode: {self.mode}")
```

## 七、agent.py 中如何使用 decision

```python
# agent.py _execute_tool()

decision = self.permission_checker.check(tool, tc.arguments)

if decision.effect == "deny":
    # 直接拒绝，返回错误 ToolResult
    yield ToolResult(output=f"Permission denied: {decision.reason}", is_error=True)
    return

if decision.effect == "ask":
    # HITL：yield PermissionRequest 事件，等待用户回应
    future = loop.create_future()
    yield PermissionRequest(tool_name=..., description=..., future=future)
    response = await future  # 阻塞直到用户确认

    if response == PermissionResponse.DENY:
        yield ToolResult(output="Permission denied: 用户拒绝了此操作", ...)
        return

    if response == PermissionResponse.ALLOW_ALWAYS:
        # 同时写入规则文件和会话内存
        self.permission_checker.add_session_allow(tool_name, content)
```

## 八、面试话术模板

> "权限系统的核心设计理念是纵深防御——不是一道门而是四道门。最外层是用户可选的全局模式，决定读/写/命令三类工具的默认行为。第二层是持久化规则引擎——用户在 YAML 里配置'永远允许 npm test'或'永远阻止读 .env'，跨会话有效。第三层是危险命令实时检测——用 pattern 匹配识别 rm -rf、curl pipe bash 等。第四层是路径沙箱——限制 Agent 只能在指定目录操作。
>
> 4 层之间的关系是：越靠近内层越具体。全局模式告诉你'写操作一般需要确认'，规则引擎告诉你'但写入 .md 文件永远允许'，危险检测告诉你'但即使是写入，rm -rf 也永远拒绝'。每层独立可测，修改一层的配置不影响其他层。"

## 九、进阶面试追问

**Q: 如果用户想在 PLAN 模式下运行 Bash，怎么办？**

A: PLAN 模式的设计目标就是"先计划，再批准执行"——不允许直接执行 Bash。用户可以在 PLAN 模式下让 Agent 写好方案，然后切换到 DEFAULT 或 BYPASS 模式执行。这是有意的限制——PLAN 模式的价值就在于安全地探索，不冒执行风险。

**Q: 权限系统会不会影响 Agent 的效率？**

A: 会——`DEFAULT` 模式下每次 write/command 操作都要用户确认。但这是安全性的代价。用户可以通过 `ACCEPT_EDITS` 模式减少编辑确认，通过规则引擎配置特定命令自动放行，在安全性和效率之间自己调节。框架不做价值判断——给用户充分的选择权。
