# 09 — 命令系统

> **面试问题**："你怎么设计一个可扩展的终端命令系统？"
> **源文件**：[devflow/commands/](../devflow/commands/) (15+命令处理器)
> **核心架构**：Parser → Registry → Handler，支持用户自定义扩展

---

## 一、命令系统架构

```
用户输入 "/compact"
    │
    ▼
CommandParser.parse("/compact")
    │
    ├── 识别："/" 前缀 → 命令模式
    ├── 解析：command="compact", args=[]
    │
    ▼
CommandRegistry.lookup("compact")
    │
    ├── 匹配：内置命令 "compact"
    │
    ▼
handle_compact(ctx) → 触发 Agent 内部方法
```

## 二、Parser —— 命令解析

```python
# parser.py
def parse_command(text: str) -> ParsedCommand | None:
    if not text.startswith("/"):
        return None  # 不是命令，当作普通对话

    parts = text[1:].strip().split()
    name = parts[0]
    args = parts[1:] if len(parts) > 1 else []
    return ParsedCommand(name=name, args=args)
```

## 三、Registry —— 命令注册

```python
@dataclass
class Command:
    name: str              # "compact", "help", "memory", ...
    description: str       # "/compact - 压缩对话上下文"
    handler: Callable      # async function
    command_type: CommandType  # BUILTIN / USER

class CommandRegistry:
    _commands: dict[str, Command]

    def register(self, command: Command): ...
    def lookup(self, name: str) -> Command | None: ...
    def list_all(self) -> list[Command]: ...
    def unregister(self, name: str): ...
```

## 五、命令如何和 Agent 交互

### 5.1 /compact 的实现

```python
# commands/handlers/compact.py
async def handle_compact(ctx: CommandContext):
    agent = ctx.agent            # 通过 CommandContext 访问 Agent
    conv = ctx.conversation      # 和 ConversationManager

    # 调用 Agent 的 internal compact 方法
    result = await agent.manual_compact(conv)

    if isinstance(result, CompactNotification):
        return result.message    # "上下文已压缩 (压缩前 45,320 tokens)"
    else:
        return result.message    # "压缩失败: ..."
```

### 5.2 CommandContext —— 注入依赖

```python
@dataclass
class CommandContext:
    agent: Agent                  # Agent 实例
    conversation: ConversationManager  # 对话管理
    work_dir: str                 # 工作目录
    permission_checker: PermissionChecker  # 权限检查器
```

**为什么用 CommandContext 而不是全局变量？** 依赖注入让命令处理器可测试——测试时传入 mock 的 agent 和 conversation，不需要启动真实的 Agent 循环。

## 六、用户自定义命令

### 6.1 加载路径

```python
# commands/loader.py
def load_user_commands(work_dir: str, registry: CommandRegistry):
    dirs = [
        Path.home() / ".devflow" / "commands",    # 用户级
        Path(work_dir) / ".devflow" / "commands", # 项目级
    ]
    for d in dirs:
        for md_file in d.glob("*.md"):
            cmd = parse_command_file(md_file)
            registry.register(cmd)
```

### 6.2 自定义命令格式

自定义命令是 Markdown 文件——包含 frontmatter 元数据和执行内容：

```markdown
---
name: deploy
description: 部署当前项目到生产环境
command_type: user
args:
  - name: env
    description: 部署环境 (prod/staging)
    required: true
---

执行以下部署步骤：
1. 运行测试: `pytest`
2. 构建镜像: `docker build -t app:latest .`
3. 推送到 registry
4. 使用 {{args.env}} 环境配置部署
```

**为什么用 Markdown 而不是 Python？**
- Markdown 可以被 LLM 阅读——自定义命令的内容会注入到 system prompt
- 降低用户门槛——不需要会写 Python
- 对齐 Claude Code 的 slash command 格式

## 七、面试话术模板

> "命令系统采用 Parser-Registry-Handler 三层架构。Parser 识别 '/' 前缀并解析命令名和参数，Registry 维护命令注册表——包括 15 个内置命令和用户自定义命令，Handler 通过 CommandContext 注入 Agent 和 Conversation 的依赖来执行具体操作。
>
> 内置命令本质上是 Agent 内部能力的暴露——/compact 触发上下文压缩、/permission 切换权限模式、/memory 管理记忆。用户自定义命令用 Markdown 文件定义——放在 ~/.devflow/commands/ 或项目 .devflow/commands/ 下，启动时自动加载。选用 Markdown 而不是 Python 是为了降低门槛——用户写一段描述就够了。
>
> 关键设计是 CommandContext 的依赖注入——每个 handler 通过 Context 获得 agent、conversation、permission_checker 的引用。这让 handler 可测试——测试时 mock Context 就行了。"

## 八、进阶面试追问

**Q: 自定义命令和 Skills 有什么区别？**

A: 命令是用户直接触发的（`/deploy`），Skill 是 LLM 按需调用的（Agent 判断当前任务需要 code-review skill 然后主动 invoke）。命令对应"用户告诉 Agent 做什么"，Skill 对应"Agent 自己判断什么时候加载什么能力"。两者的关系是互补的——命令改变 Agent 的外部行为，Skill 扩展 Agent 的内部能力。
