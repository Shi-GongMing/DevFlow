# 07 — 记忆系统

> **面试问题**："Agent 怎么记住用户偏好和项目上下文？"
> **源文件**：[devflow/memory/](../devflow/memory/) (5个文件)
> **核心机制**：自动提取 + 文件存储 + 会话召回

---

## 一、什么时候该"记住"什么？

记忆系统的核心挑战不是"怎么存"，而是"什么时候记住什么"：
- 记得太频繁：浪费 LLM 调用 token
- 记得太稀疏：漏掉重要信息
- 记得太多：MEMORY.md 膨胀，召回质量下降

---

## 二、自动记忆提取（Auto Memory）

### 2.1 触发时机

```python
# agent.py
MEMORY_EXTRACTION_INTERVAL = 5  # 每 5 轮对话触发一次

# 在主循环中
if self._loop_count % MEMORY_EXTRACTION_INTERVAL == 0 and self.memory_manager:
    asyncio.ensure_future(self._extract_memories(conversation))
```

**为什么是 5 轮而不是每轮？** 每轮提取浪费 token（对话太短没有有价值的记忆），间隔太大漏信息。5 轮是经验平衡点——通常 5 轮对话后用户会表现出一些稳定偏好或项目上下文值得记住。

### 2.2 合并策略（inProgress + pendingContext）

```python
async def _extract_memories(self, conversation):
    # 如果正在提取，暂存新请求
    if self._extracting:
        self._pending_extraction = True
        return

    self._extracting = True
    try:
        await self.memory_manager.extract(client, conversation, protocol)
    finally:
        self._extracting = False
        # 检查是否有尾随请求
        if self._pending_extraction:
            self._pending_extraction = False
            await self._extract_memories(conversation)  # 尾随提取
```

**为什么需要合并策略？** 提取操作异步 (`asyncio.ensure_future`)，如果 5 轮后立即又触发了一次（第 6 轮恰好在提取完成前），不合并会导致两次并发提取——两次提取看到几乎相同的对话，产生重复记忆。合并策略确保最多只有一个提取在进行。

## 三、存储格式

### 3.1 独立文件 + Frontmatter + 索引

每条记忆是一个独立的 Markdown 文件：

```markdown
---
name: user-prefers-chinese
description: 用户偏好中文回复和中文注释
metadata:
  type: user
---

用户明确要求所有回复使用中文，代码注释也使用中文。
```

**为什么用独立文件而不是单文件或数据库？**

```
方案 A（DevFlow 选的）：独立 .md 文件 + MEMORY.md 索引
方案 B1：单文件集中存储（一个 memories.json）
方案 B2：SQLite 数据库
```

**A 比 B1（单文件）好在哪：**
- 每条记忆独立——不会因为修改一条记忆导致整个文件被重写
- 可以被 git 追踪（项目级记忆）
- 人类可以直接阅读和编辑

**A 比 B2（SQLite）好在哪：**
- 不需要 SQL 查询——LLM 可以直接阅读 MEMORY.md 索引
- 跨语言——任何语言都能读写 markdown 文件
- git diff 友好

**A 的代价：**
- 文件数量多时（100+ 条）查询性能差
- 但记忆数量通常不超过 50 条——全量加载完全可行

### 3.2 MEMORY.md 索引

```markdown
- [用户偏好中文](user-prefers-chinese.md) — 回复和注释使用中文
- [项目测试框架](project-test-framework.md) — 使用 pytest + pytest-asyncio
- [代码规范](project-code-style.md) — snake_case, commit 用英文
```

```python
MAX_ENTRYPOINT_LINES = 200    # MEMORY.md 最多 200 行
MAX_ENTRYPOINT_BYTES = 25_000 # 最多 25KB
# 超过截断——防止索引文件膨胀
```

### 3.3 两级存储目录

```python
# 用户级 (~/.devflow/memory/)：user 和 feedback 类型
_USER_LEVEL_TYPES = {"user", "feedback"}

# 项目级 (<project>/.devflow/memory/)：project 和 reference 类型
_PROJECT_LEVEL_TYPES = {"project", "reference"}
```

| 记忆类型 | 存储位置 | 适用范围 | 示例 |
|----------|----------|----------|------|
| user | 用户级 | 跨所有项目 | "用中文回复" |
| feedback | 用户级 | 跨所有项目 | "不要 mock 数据库" |
| project | 项目级 | 当前项目 | "测试框架是 pytest" |
| reference | 项目级 | 当前项目 | "Bug 跟踪用 Linear" |

**为什么分两级？** 用户偏好跨项目有效（你在所有项目里都偏好中文），项目上下文只在当前项目有效（这个 repo 的 CI 用 GitHub Actions，那个 repo 用 Jenkins）。

## 四、记忆召回（Recall）

### 4.1 会话启动时的召回

```python
# agent.py run()
# 1. 加载 instructions (MEWCODE.md + AGENTS.md)
instructions_content = load_instructions(work_dir)

# 2. 加载长期记忆
memory_content = self.memory_manager.load() if self.memory_manager else ""

# 3. 注入到对话开头
conversation.inject_long_term_memory(instructions_content, memory_content)
```

### 4.2 记忆选择策略

```python
# recall.py
# LLM 从 MEMORY.md 索引中选择相关记忆（最多 5 条）
prompt = """You are selecting memories that will be useful to DevFlow
as it processes the user's query (up to 5). Only include memories
that are usage reference or API documentation for those tools."""
```

**为什么用 LLM 选而不是全文注入？** MEMORY.md 可能有几十条记忆，全部注入会吃掉大量 system prompt token。LLM 选择只挑最相关的 1-5 条——信息密度最高。

## 五、Env 覆盖

```python
# auto_memory.py
# 远程模式下可以用环境变量覆盖记忆目录
override = os.environ.get("DEVFLOW_REMOTE_MEMORY_DIR", "")
if override:
    return override + os.sep
return os.path.join(abs_root, ".devflow", "memory") + os.sep
```

**为什么需要环境变量覆盖？** DevFlow 的 remote 模式在浏览器中运行——此时 `work_dir` 可能是服务器上的路径。环境变量让运维可以指定集中式的记忆存储位置。

## 六、面试话术模板

> "记忆系统的核心挑战是'什么时候该记住什么'。我的方案是每 5 轮对话自动触发一次记忆提取——LLM 回顾近期对话，判断是否有值得持久化的信息，用独立 markdown 文件加 frontmatter 元数据存储。MEMORY.md 作为索引，LLM 在提取时先读索引判断是否已有类似记忆、需要更新还是新增。
>
> 存储分两级：用户偏好和反馈纠正放在 ~/.devflow/memory/——跨项目有效；项目上下文和引用放在项目 .devflow/memory/——只在当前项目有效。选择 markdown 文件而不是数据库的理由是——人类可编辑、git 可追踪、LLM 可直接阅读。代价是查询性能随文件数线性下降，但记忆数量通常不超过 50 条，全量加载完全可行。
>
> 一个重要的工程细节是合并策略——用 inProgress + pendingContext 防止两个异步提取并发执行导致重复记忆。"

## 七、进阶面试追问

**Q: 如果 LLM 提取了错误或过时的记忆怎么办？**

A: 每个记忆是独立的 markdown 文件——用户可以手动编辑或删除。MEMORY.md 索引也支持手动改。如果错误记忆进入了系统，下一次提取时 LLM 会对比已有记忆和当前对话——如果发现矛盾，它可能选择更新或删除。但这个"自我纠错"能力不是 100% 可靠的——最终的安全网是用户的手动干预。

**Q: 和 LangChain 的 ConversationBufferMemory 有什么区别？**

A: `ConversationBufferMemory` 是会话级短暂记忆——缓存最近的对话轮次，下一次会话不保留。DevFlow 的记忆系统是跨会话持久化记忆——提取的是用户偏好、项目约束、技术决策这些长期有价值的信息。两者不是替代关系而是互补——短暂记忆保证当前对话连贯，持久记忆保证跨会话的知识积累。
