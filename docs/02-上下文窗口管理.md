# 02 — 上下文窗口管理

> **面试问题**："LLM 的 context window 有限，长对话中你怎么管理上下文？"
> **源文件**：[devflow/context/manager.py](../devflow/context/manager.py) (842行)
> **依赖关系**：被 agent.py 在每轮迭代开始时调用，依赖 conversation.py 和 serialization.py

---

## 一、为什么这是 Agent 框架最核心的问题

LLM 的上下文窗口是硬限制。超出窗口 → API 返回错误 → Agent 直接挂掉。所以上下文管理不是"优化"，而是**必须**——没有它，Agent 无法进行长对话。

但上下文管理的挑战不止是"不要超过窗口"：
- 在窗口中放什么信息？放越多历史越连贯，但留给工具结果和思考的空间越小
- 什么时候触发处理？太早浪费 token，太晚 API 报错
- 怎么处理？截断最简单但丢信息，摘要保留语义但消耗额外 token

DevFlow 用 **两层管理 + 熔断器** 回答了这三个问题。

---

## 二、两层上下文管理

### Layer 1：工具结果预算（Tool Result Budget）

**问题**：工具输出可能很大（`grep` 返回 50K 文本、`cat` 读 30K 文件），直接塞入对话会吃掉大量 context window。

**方案**：三遍扫描替换机制。

**核心常量**：
```python
SINGLE_RESULT_CHAR_LIMIT = 50_000    # 单条结果超过 50K → 落盘
AGGREGATE_CHAR_LIMIT = 200_000       # 本轮汇总超过 200K → 从大到小落盘
PREVIEW_CHARS = 2_000                # 落盘后保留的预览字符数
MAX_OUTPUT_CHARS = 10_000            # 工具输出的硬截断
```

**三遍扫描算法** (`apply_tool_result_budget`)：

```
Phase 1: 逐个检查
  for each tool_result:
    if len(content) > 50_000:
      写入磁盘 (.devflow/session/tool-results/{tool_use_id}.txt)
      替换为 2KB 预览 + 文件路径
      mark as "seen"

Phase 2: 汇总检查
  如果本轮所有结果总和 > 200_000:
    从大到小排序, 超出阈值的全部落盘

Phase 3: 标记未替换的 fresh 结果为 "seen"
```

**落盘后的预览格式**：
```
<persisted-output>
输出太大（523KB），完整内容已保存到：
/Users/.../.devflow/session/tool-results/toolu_xxx.txt

预览（前 2KB）：
... 前 2000 字符 ...
</persisted-output>
```

**关键设计：Design B（不修改原对话）**：

```python
def apply_tool_result_budget(conversation, session_dir, state):
    # 返回一个 NEW ConversationManager，不修改原始 conversation
    new_conv = ConversationManager()
    new_conv.history = new_history  # 新历史，有替换后的 tool_results
    return new_conv, new_records
```

为什么？原始 conversation 用于持久化（保存完整内容），副本用于 LLM 调用（有替换后的精简内容）。两个用途需要不同的数据，直接修改原始会导致数据丢失。

### Layer 2：全对话摘要压缩（Auto-Compact）

**问题**：对话越长，历史消息积累越多。超过一定阈值后，即使 tool results 都优化了，纯对话也塞不下。

**方案**：双阈值触发 + 前缀摘要 + 尾部保留。

**触发策略**：
```python
AUTO_COMPACT_SAFETY_MARGIN = 13_000   # 软触发：context_window - 20K - 13K
MANUAL_COMPACT_SAFETY_MARGIN = 3_000  # 硬触发：context_window - 20K - 3K

effective_window = context_window - SUMMARY_OUTPUT_RESERVE(20K)

# 软触发线：effective - 13K
# token 超过软触发线 → 走熔断器检查 → 执行压缩

# 硬触发线：effective - 3K
# token 超过硬触发线 → 跳过熔断器 → 强制压缩
```

**为什么是双阈值？**

```
软触发（-13K）：还有缓冲空间 → 走完整流程（包括熔断器保护）
硬触发（-3K）： 快到天花板了 → 跳过熔断器，无论如何必须压缩
```

单阈值的问题：如果阈值设太高（-3K），正常压缩被推迟，到了临界线才急急忙忙做；如果设太低（-20K），频繁压缩浪费摘要 token。双阈值在"及时压缩"和"不过度压缩"之间取了平衡。

**保留窗口策略** (`_compute_keep_start_index`)：

```python
KEEP_RECENT_TOKENS = 10_000    # 最近 10K token 原文保留
MIN_KEEP_MESSAGES = 5          # 至少保留 5 条 （取先满足的条件）
KEEP_MAX_TOKENS = 40_000       # 单条消息拖入窗口的上限

# 从尾部向头部遍历：
# - 累计 token < 10K OR 保留数 < 5 → 继续纳入保留窗口
# - 但下一次纳入会超过 40K → 立即停止（防单条吞掉整个窗口）
```

**配对保护** (`_align_keep_start_to_tool_pair`)：

```python
# 如果 keep_start 落在 tool_result 消息上，
# 向前回退到配对的 assistant(tool_uses) 消息
# 避免"只有 tool_result 没有 tool_use"的半条消息
```

**前缀太小不值得压缩**：
```python
MIN_SUMMARIZE_PREFIX_TOKENS = 2_000
# 前缀 < 2K token 时不压缩——摘要消耗的 token 比回收的还多
```

**摘要 Prompt 设计**：

```python
SUMMARY_PROMPT = """\
你先在 <analysis> 标签中分析（会被丢弃），
然后在 <summary> 标签中输出正式摘要。

<summary> 必须包含 9 个部分：
1. 主要请求和意图
2. 关键技术概念
3. 文件和代码段（关键代码保留）
4. 错误和修复
5. 问题解决过程
6. 所有用户消息（原文保留，不可改写！）
7. 待办任务
8. 当前工作（最详细）
9. 可能的下一步"""
```

**为什么用 9 部分结构化摘要而不是让 LLM 自由发挥？** 自由摘要倾向于保留"叙述"而丢弃"细节"——第 6 部分强制保留所有用户消息原文、第 3 部分保留关键代码片段、第 4 部分保留具体错误。这些是后续对话最需要的信息。

**压缩后的对话重建**：

```python
# 压缩前：[msg1, msg2, ..., msg50, msg51, ..., msg60]
#                     ↑ prefix (摘要)     ↑ keep_tail (保留)
# 压缩后：[摘要(user消息), msg51, ..., msg60]

new_messages = build_compact_messages(summary, ...)
               + list(keep_tail)
conversation.replace_history(new_messages)
# 清空用量锚点，下一轮 API 调用会重新锚定
```

### 2.3 压缩后恢复（Recovery）

压缩后 Agent 的对话历史被清空了——模型会"忘记"刚才读过哪些文件、正在执行哪个 skill。`RecoveryState` 解决了这个问题：

```python
@dataclass
class RecoveryState:
    _files: dict[str, FileReadRecord]   # 最近读过的文件内容快照
    _skills: dict[str, SkillInvocationRecord]  # 激活的 skill 正文

# agent.py 中每次 ReadFile 后:
def _snapshot_for_recovery(self, tc, result):
    if tc.tool_name == "ReadFile" and not result.is_error:
        self.recovery_state.record_file_read(path, content)
```

**压缩后附件会追加到摘要消息中**：
```
## 最近读过的文件
### src/agent.py (read 2026-06-26T10:30:00Z)
```python
...文件内容（最多5000 token）...
```

## 已激活的技能
### code-review
... skill 正文 ...

## 可用工具
- Bash — 执行 shell 命令
...

## 提示
以上恢复的上下文是重建的。若需要原文代码，请用文件读取工具重新读取。
```

### 2.4 熔断器（Circuit Breaker）

```python
@dataclass
class CompactCircuitBreaker:
    max_failures: int = 3
    consecutive_failures: int = 0

    def is_open(self) -> bool:
        return self.consecutive_failures >= self.max_failures
```

**为什么需要熔断器？** 摘要操作本身消耗 token 且可能失败——如果 LLM 连续 3 次摘要失败，继续重试只会浪费更多 token。熔断器打开后拒绝自动压缩，让用户手动 `/compact`。

**但硬触发时绕过熔断器**——因为硬触发时已经接近窗口上限，不压缩就会 API 报错，两害相权取其轻。

## 三、Token 估算策略

```python
# UsageAnchor：记录最后一次真实 API 用量
@dataclass
class UsageAnchor:
    baseline_tokens: int  # input + cache_read + cache_creation + output
    anchor_count: int     # 当时的消息数
    has_usage: bool       # 是否收到过 API 用量报告

# current_tokens() = baseline_tokens + 锚点后新增消息的字符估算
# 冷启动 (has_usage=False) → 退化到全量字符估算
```

**为什么用真实 API 用量做锚点而不是纯字符估算？** 字符估算是近似的（中文 1 字符 ≈ 0.5 token，英文 1 字符 ≈ 0.25 token），误差会累积。混合策略在准确性和计算成本之间取平衡——有真实数据时用真实数据，没有时退化为估算。

## 四、面试话术模板

> "上下文管理是 Agent 框架的核心挑战。我的方案是两层管理加熔断器保护。
>
> 第一层是工具结果预算——三遍扫描算法：单条超 50K 字符的工具结果落盘并用 2K 预览替换，全部结果超 200K 时从大到小继续落盘。关键设计是用 Design B——不修改原始对话，而是生成副本给 LLM 用，原始对话保留完整数据用于持久化。
>
> 第二层是全对话摘要压缩——双阈值触发：软触发在距窗口 13K token 时走熔断器检查，硬触发在距 3K 时强制压缩。压缩时保留最近 10K token 的原文保证连贯性，对更早的消息生成包含 9 个部分的结构化摘要。压缩后有恢复附件——保存了最近读过的文件和激活的 skill，让模型不丢失工作上下文。
>
> 熔断器的设计参考了微服务架构的 circuit breaker 模式——连续 3 次压缩失败后拒绝自动压缩，防止浪费 token。但硬触发时绕过熔断器，因为 API 调用失败比浪费 token 更糟。"

## 五、进阶面试追问

**Q: 为什么不直接用 LangChain 的 trim_messages？**

A: `trim_messages` 只是简单丢弃旧消息——基于 token 数或消息数的粗暴截断。它不生成摘要，不保留原文尾部，不会有恢复附件。用 `trim_messages` 的话，Agent 在压缩后会完全丢失旧对话的上下文。DevFlow 的方案在信息保留和 token 效率之间取了平衡——代价是实现复杂度高了一个数量级（842 行 vs LangChain 的 20 行）。

**Q: 摘要模型出错了怎么办？**

A: 摘要用和主对话相同的 LLM 模型。如果 LLM 返回的不是期望的 `<summary>` 格式，`extract_summary()` 回退到使用全部 LLM 输出。如果 LLM 连续失败 3 次，熔断器打开——防止无限重试浪费 token。用户可以手动 `/compact`。
