# 03 — 工具系统设计

> **面试问题**："你会怎么给 Agent 设计一个安全可扩展的工具系统？"
> **源文件**：[devflow/tools/](../devflow/tools/) (15+工具实现)
> **核心依赖**：tools/base.py → permissions/checker.py（权限门控）→ agent.py（注册和调用）

---

## 一、工具系统的架构

```
Tool (ABC)  ← 所有工具的基类
  ├── name: str              # "Bash", "ReadFile", ...
  ├── description: str       # LLM 看到的工具描述
  ├── params_model: BaseModel # Pydantic 参数模型
  ├── category: ToolCategory  # read / write / command
  ├── is_concurrency_safe    # 能否并行执行
  ├── is_system_tool         # 系统工具（不在 UI 展示）
  ├── should_defer           # 延迟加载（需 ToolSearch）
  ├── schema() → dict        # 生成 LLM function schema
  └── execute(params) → ToolResult  # 核心：执行工具

ToolRegistry  ← 工具注册与查询
  ├── register(tool)
  ├── get(name) → Tool
  ├── is_enabled(name) → bool
  ├── get_all_schemas(protocol) → list[dict]
  └── get_deferred_tool_names() → list[str]
```

## 二、Base Tool 类的 6 个字段解析

### 2.1 name + description —— LLM 看到的

```python
class Bash(Tool):
    name = "Bash"
    description = "执行 shell 命令。使用 jq、curl 等标准工具..."
```

这两个字段直接进入 LLM 的 system prompt（作为 tool schema 的一部分）。`description` 的质量直接影响 LLM 调用的正确率。

### 2.2 params_model —— Pydantic 参数校验

```python
class BashParams(BaseModel):
    command: str = Field(description="要执行的命令")
    timeout: int = Field(default=120000, description="超时毫秒")

class Bash(Tool):
    params_model = BashParams

    async def execute(self, params: BashParams) -> ToolResult:
        # params 已经被 Pydantic 校验过
        result = await run_command(params.command, timeout=params.timeout)
        return ToolResult(output=result, is_error=False)
```

**为什么用 Pydantic 而不是手动校验？**
- Pydantic 的 `model_json_schema()` 能自动生成 JSON Schema——比手写准确、和 Anthropic/OpenAI 的 tool schema 格式一致
- `model_validate()` 在工具执行前做参数校验——如果 LLM 传了错误参数，错误在工具执行前被捕获
- 字段的 `Field(description=...)` 进入 schema，LLM 能看到参数说明

### 2.3 category —— 权限模式的核心

```python
ToolCategory = Literal["read", "write", "command"]

class Bash(Tool):
    category = "command"      # 需要权限检查

class ReadFile(Tool):
    category = "read"         # default 模式下自动放行

class WriteFile(Tool):
    category = "write"        # default 模式下需要询问
```

category 决定了 `mode_decide()` 的结果：

| category | DEFAULT | ACCEPT_EDITS | PLAN | BYPASS |
|----------|---------|-------------|------|--------|
| read | allow | allow | allow | allow |
| write | ask | allow | ask | allow |
| command | ask | ask | ask | allow |

### 2.4 is_concurrency_safe —— 并行执行

```python
# agent.py 中的 partition_tool_calls()
# 连续的 is_concurrency_safe=True 工具 → 并行 batch
# is_concurrency_safe=False 的工具 → 单独串行 batch
```

哪些工具是并发安全的？`ReadFile`、`Glob`（纯读取）。哪些不是？`Bash`（可能改变系统状态）、`EditFile`（可能有写入冲突）。

### 2.5 should_defer —— 延迟加载

```python
# 工具太多时，有些 schema 不随 system prompt 发送
# 只在 ToolSearch 工具中暴露搜索入口
# LLM 需要时通过 ToolSearch 加载具体 schema
```

**为什么需要延迟加载？** 工具 schema 也占 token。如果有 100+ 个 MCP 工具，全部 schema 可能数千 token。延迟加载只发送内置工具的 schema，外部工具通过 ToolSearch 按需加载。

## 四、工具输出的三层截断

```python
# agent.py _maybe_persist_or_truncate()

def _maybe_persist_or_truncate(self, tool_use_id, text):
    if len(text) > 50_000:        # 单条超限
        fp = persist_tool_result(tool_use_id, text)
        return make_persisted_preview(text, fp)    # 落盘+预览
    if len(text) > 10_000:        # 硬截断
        return text[:10_000] + "\n… (output truncated)"
    return text                    # 完整保留
```

三层逻辑：
1. **> 50K 字符**：写磁盘，返回 2K 预览 + 文件路径（LLM 可后续读取完整文件）
2. **10K ~ 50K**：截断到 10K，末尾加截断标记
3. **< 10K**：完整保留

## 五、流式事件系统（和工具的衔接）

```python
# LLM 的流式响应中，tool call 是增量到达的
@dataclass
class ToolCallStart:       # "Bash" tool call 开始
    tool_name: str
    tool_id: str

@dataclass
class ToolCallDelta:       # 参数片段 "ls -l" → "/tmp" → ...
    text: str               # JSON 增量

@dataclass
class ToolCallComplete:     # tool call 完整接收
    tool_id: str
    tool_name: str
    arguments: dict[str, Any]   # 完整 JSON → dict
```

`StreamCollector` 不产出 `ToolCallStart` / `ToolCallDelta` 事件（pass），只在 `ToolCallComplete` 时产出 `ToolUseEvent`——其他消费者只需要完整的 tool call，不关心增量过程。

## 六、面试话术模板

> "工具系统的核心是 Tool 抽象基类。每个工具有 6 个关键属性：name 和 description 决定了 LLM 如何理解和使用工具；params_model（Pydantic）既自动生成 JSON Schema 给 LLM 看，又在执行前做参数校验；category 决定权限模式下的默认行为——read 在 default 模式放行，write 和 command 需要确认；is_concurrency_safe 控制并行还是串行执行；should_defer 支持延迟加载，减少 system prompt 的 token 消耗。
>
> 工具输出有三层截断：超过 50K 字符落盘用预览替换，10K 到 50K 之间硬截断，10K 以内完整保留。这样 Agent 既能看到关键信息，又不会一个工具调用就吃掉整个 context window。"

## 七、进阶面试追问

**Q: 为什么不直接用 function calling schema 定义？**

A: Anthropic 和 OpenAI 的 tool schema 格式不同——Anthropic 用 `input_schema`，OpenAI 用 `parameters` + `function` 包装。DevFlow 的 Tool 基类统一了接口——`get_schema()` 返回标准化的 dict，`client.py` 按协议类型做适配。这样新增工具时不需要关心协议差异。

**Q: 如何处理工具执行中的异常？**

A: 三层保护：Pydantic 校验在参数层拦截（`ValidationError` → `ToolResult(is_error=True)`），execute 内部异常被 catch 并返回 error result，不会让 Agent 主循环崩溃。如果连续 3 次调用未知工具（LLM 幻觉），Agent 终止——这是防御性安全网。
